ACP agents should reuse their own subscriptions while appearing as native VS Code agents.

本报告重新锚定目标：支持 ACP 的 agent 全部复用自身已有订阅、登录、计费和模型路由；VS Code 负责把交互、状态和显示适配到现有 AgentHost / Agent Sessions UI。

执行结论

目标重新锚定。 已安装、已支持 ACP 的 agent 可以作为 VS Code AgentHost 中的第三方可选 agent 出现在 agent 列表里。所有这些 agent 都复用自己的订阅、登录、计费和模型路由；VS Code 不替它们做模型 Provider broker。

难点转移到 UI 语义适配。 不再把通用 Director Provider 切换作为本阶段目标。真正的难点是把 ACP 的 session/update、auth-required、模型/模式、工具调用、权限、文件编辑、终端输出和日志诊断，映射到 VS Code 已有 UI，而不是另做一个 ACP Webview。

参考项目与文档都是什么

ACP 官方协议与 registry

ACP 是 editor-to-agent 协议，不是 LLM Provider API。它定义初始化、session/new、session/prompt、session/update、认证、文件/终端/权限请求等 editor 与 agent process 的交互。

对我们的参考 协议生命周期、能力协商、auth-required 重试、registry 元数据、以及安全边界。

`vscode-acp`

这是一个 VS Code extension，用 extension host + Activity Bar + Tree View + Chat Webview 连接 ACP agent。它有 process spawn、stdio JSON-RPC、auth retry、file/terminal/permission handlers。

对我们的参考 适合借鉴 transport/lifecycle，不适合照搬 UI；我们要融入 AgentHost UI。

Zed

Zed 是最接近产品化的参考：registry/settings 产生 agent command，connection store 启动 ACP subprocess，ACP thread adapter 把更新映射到编辑器会话 UI。

对我们的参考 Workbench 管理配置/registry，AgentHost node 管 runtime，UI 使用既有 agent shell。

Cursor Agent ACP

Cursor Agent 通过 `agent acp` 或 `cursor-agent acp` 作为 ACP server 运行。订阅、登录和模型路由由 Cursor 自己负责。

对我们的参考 可作为外部 ACP agent；VS Code 要显示“使用 Cursor 订阅/登录”。

CodeBuddy Code ACP

CodeBuddy 通过 `codebuddy --acp` 运行，支持自己的登录、API key、站点/企业环境和模型服务配置。

对我们的参考 可作为 CodeBuddy-owned agent 接入；VS Code 不接管其订阅。

当前 VS Code fork

当前 fork 已有 `IAgent`、AgentHost root state、Agent Sessions UI、Director Agent、Director Provider Backend Hub、Director runtime credential bridge。

对我们的参考 扩展点已经存在；关键是新建 ACP runtime adapter。

推荐架构

ACP 的产品边界应是“外部 agent runtime + vendor-owned subscription”，不是“模型 Provider 后端”。推荐结构如下：

Workbench ACP Settings / Registry UI
        |
secret-free ACP agent config snapshot
        |
AgentHost node ACP agent manager
        |
one AcpAgent provider per enabled ACP agent
        |
ACP stdio JSON-RPC process connection
        |
external ACP agent process
        |
vendor-owned subscription/login/model/runtime

这条路径应和 Director 保持并行：

Director AgentHost provider
        |
Director AgentEngine adapter
        |
Director Provider Backend Hub
        |
Director API key / OAuth / local / compatible Providers

为什么这样融入 UI

当前 `IAgent` provider 会被 AgentHost side effects 发布为 root `AgentInfo`，而 Agent Sessions provider 会把 root `AgentInfo` 映射成 UI 可选 session type。因此，只要 ACP agent 以 `IAgent` provider 注册，就能自然进入现有 agent 列表，不需要新建 ACP Activity Bar 或 Webview。

阶段 Roadmap

完整 roadmap 已单独落到 06-acp-agenthost-roadmap.md。阶段拆分如下：

阶段 名称 主要产出 难度
0 Boundary And UX Contract 确认 ACP agents 复用自身订阅，确认非目标和 UI 文案边界。 低
1 Manual ACP Agent Configuration 提供基础 External ACP Agents 管理页，支持手动 add/edit/remove/enable/trust，并生成 secret-free snapshot。 低到中
2 ACP Runtime Skeleton AgentHost node 可启动、initialize、timeout、dispose fake/manual ACP process。 中
3 Agent List And Subscription State 已启用 ACP agents 出现在 VS Code agent 列表，并在 session 创建或 agent detail 路径显示订阅归属；列表展示不得 eager 启动外部命令。 中
4 Basic Text Session 预登录 ACP agent 可创建 session、发送 prompt、在 Agent Sessions UI 看见流式文本/思考输出；未登录时给出清晰提示。 中到高
5 Vendor Login UX And Smoke Tests 把未登录提示升级为 guided recovery，并对 Cursor/CodeBuddy 等真实命令做 smoke test。 中到高
6 Tools, Permissions, Files, Terminal ACP side effects 全部经由 AgentHost 原生权限、工具、文件、终端 UI。 高
7 Registry Browse And Managed Enablement 支持 registry 浏览、信任门禁、版本 pin、安装/启用策略。 高/安全
8 Models, Modes, Config, Restore 按 capability 显示 models/modes/config/restore，不做能力假设。 高
9 Hardening And Policy Windows/remote、policy、redacted diagnostics、最终文档和 release gate。 高

推荐第一里程碑：只做 Phase 0-4，也就是 manual ACP command → AgentHost provider appears in UI → pre-authenticated session → send text prompt → stream text → cancel/dispose cleanly。若真实 vendor agent 未登录，只要求给出清晰的 vendor-login-required 提示。

Phase 4-6 review update. Phase 3/4 必须从一开始实现完整 `IAgent` skeleton，未到阶段的方法要有明确 unsupported/no-op 语义。Phase 4 的 text-only turn 要把意外 `tool_call` / `tool_call_update` 降级为清晰的 unsupported-capability 结果，并保证 complete/cancel/error 只发出一个终态。Phase 5 只做可控的 vendor login recovery 和 smoke test，terminal-auth 后移到 Phase 6 或 Phase 6.5。Phase 6 中 file/terminal/tool capability 必须在 ACP `initialize` 前确定，后续变更需要 reconnect/re-initialize。

Phase 7-9 review update. Phase 7A 第一版只做 registry browse、disabled draft、Phase 1 manual enablement path 和 policy/schema；Phase 7B managed install 保留为 deferred design，必须先满足 trust、policy、integrity/pin 和 local/remote install placement。Phase 8 暴露 restore UI 前必须完成 transcript 来源可信度和 VS Code session URI、ACP session id、vendor restore id 映射的 feasibility。Phase 9 不应第一次添加高风险 policy，而是验证早期阶段已接入的 policy、local placement 和 diagnostics redaction schema。

Recorded decisions. Phase 7 第一版只做 browse/manual-config-only，不自动安装 ACP agent，也不启用任何 managed-install distribution type；unverifiable registry entry 只能生成 disabled draft/manual config。Phase 8 将 restore 作为目标保留，但先做实现难度和 vendor compatibility audit，再按 vendor-gated 方式逐步开放。

Cross-phase corrections. Phase 0 已记录首版使用 internal JSON-RPC/local DTOs，SDK adoption 后置到 dependency/bundling review；Phase 6 负责 file/terminal/tool policy hooks，Phase 9 只做验证；Phase 8A 可在 Phase 5 后做 models/config 与 restore feasibility，restore/reconnect 相关实现依赖 Phase 6。

Release matrix decision. 第一版只承诺 local desktop + 本机已安装/已登录 ACP CLI：local VS Code desktop、local workspace、local AgentHost node process、local config/cache/logs、本机预安装 ACP CLI，以及该 CLI 自己已有的 vendor login/subscription。SSH/remote server、WSL、dev containers、Codespaces 等远程目标不作为第一版 release blocker。

Internal development decisions. 产品完成前按内部开发/测试处理：External ACP AgentHost support 默认开启，不做 experiment gate；ACP process/runtime telemetry 不上报；delete/archive 默认只影响 VS Code 本地记录，vendor-side delete 只能作为后续单独批准能力。

实现难点与风险

最高风险：side effects

ACP agent 可以请求文件读写和终端执行。必须经过权限、workspace containment、日志脱敏、取消清理、企业策略。

供应链风险

Registry entry 可能指向 binary、NPX、UVX。不能静默安装执行；需要版本 pin、缓存边界、信任 UI、校验策略。

协议映射风险

ACP session/update 的顺序、tool call 状态、permission request 和 cancellation 要严谨映射，否则 UI 会重复、丢状态或挂起。

订阅语义风险

用户容易把 Director Providers 和外部 agent 自己的订阅混淆。UI 必须明确账户、模型、计费和能力归属。

Windows/remote 风险

`.cmd`、PATH、shell quoting、workspace cwd、remote AgentHost process 位置都会影响 agent 是否能启动和加载正确项目配置。

版本漂移风险

Cursor/CodeBuddy 等 agent 的 session restore、model switch、permission/MCP 支持可能随版本变化。必须 capability-gate。

建议的第一张实现票

下一阶段最值得做的是一个小而完整的 prototype，验证“外部订阅 + 原生 VS Code UI”的路径：

Add a gated manual-command ACP AgentHost provider prototype:
1. register one configured ACP agent as an AgentHost provider;
2. launch it over stdio;
3. initialize ACP v1;
4. create a session with workspace cwd;
5. send one text prompt;
6. stream text into AgentHost UI;
7. cancel and dispose cleanly;
8. show that the agent uses its own subscription/login;
9. if missing login, show a clear vendor-login-required message;
10. do not advertise file/terminal capabilities yet;
11. downgrade unexpected tool updates instead of hanging the turn.

这个 slice 能证明架构，不会过早进入完整登录编排、registry 安装、工具执行、终端代理或 Director Provider 切换这些高风险区。
